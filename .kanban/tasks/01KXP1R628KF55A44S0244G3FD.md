## What

Port grok/codex apply_patch's pass-4 matcher behavior (see `xai-org/grok-build`, `seek_sequence.rs`) into our recovery ladder: an ASCII-authored `find` must match source text containing typographic punctuation (en/em dashes, smart quotes, exotic spaces) ‚Äî the single most common real-world edit-match failure. Two facets, one concern: fold confusables when matching, and name confusables when a near-miss diff still fails.

**1. `Sources/FileTool/EditMatch.swift` ‚Äî new ladder rung**

- Add a static `foldConfusables(_ text: String) -> String` that maps, per Unicode scalar:
  - dashes `U+2010`‚Äì`U+2015`, `U+2212` ‚Üí `-`
  - single quotes `U+2018`‚Äì`U+201B` ‚Üí `'`
  - double quotes `U+201C`‚Äì`U+201F` ‚Üí `"`
  - spaces `U+00A0`, `U+2000`‚Äì`U+200A`, `U+202F`, `U+205F`, `U+3000` ‚Üí space
  - everything else unchanged. Make it `package`/`public` visible ‚Äî `EditEngine` needs it for the diagnostic facet.
- Add `Rung.unicode`. Ladder order becomes exact ‚Üí normalized ‚Üí **unicode** ‚Üí anchor ‚Üí fuzzy (order is the contract; the new rung must only fire when `normalized` has descended).
- Implement the rung by generalizing the existing `locateLineBlock` line-comparison machinery to take a line normalizer, so the unicode rung is `locateLineBlock` with `normalize ‚àò foldConfusables` ‚Äî do not duplicate the window-scan loop. Uniqueness/ambiguity flows through the existing `finalizeBlockMatches` (one match ‚Üí `.unique(range:rung:confidence: 1.0)`, several ‚Üí `.ambiguous`, none ‚Üí descend).
- Do **not** change the exact, normalized, anchor, or fuzzy rungs (fuzzy scoring stays confusable-blind): the Rust-parity golden vectors in `Tests/FileToolTests/Fixtures/edit-match-golden.json` must pass byte-identical. Update the `EditMatch` header doc: the ladder is now five rungs, and the unicode rung is a deliberate extension beyond the Rust `swissarmyhammer-edit-match` crate (flag for future upstream parity).

**2. `Sources/FileTool/EditEngine.swift` ‚Äî confusable diagnostic in near-miss diffs**

When the ladder still fails (the span differs by more than punctuation), the near-miss diff currently shows a smart-quote line and an ASCII-quote line as an opaque expected/actual pair. In the diff construction (`diff(expected:actual:)` / `nearMiss(for:span:)`): when an `expected` line and an `actual` line differ but are equal after `normalize` + `EditMatch.foldConfusables`, surface a diagnostic. Add an optional `note: String?` to `EditEngine.NearMiss` (default `nil`) naming the first confusable difference with its code point, e.g. `differs only by Unicode punctuation: the file has '\u{2019}' (U+2019) where the find has "'" (U+0027)`.

**3. `Sources/FileTool/Operations/EditFile.swift` ‚Äî wire projection**

Project the new note onto `EditNearMiss` as an optional `note` field (nil-omitted like the other optional outcome fields). No status/`matchedBy` changes: a unicode-rung match already surfaces as `recovered` via the existing `EditEngine.Resolution` mapping.

## Acceptance Criteria

- [ ] A `find` written with ASCII `-`, `'`, `"`, and plain spaces resolves to a `.unique` match with rung `.unicode` and confidence `1.0` against content containing en/em dashes, smart quotes, and no-break/thin spaces, with the returned byte range indexing the original (typographic) bytes.
- [ ] Several confusable-folded occurrences yield `.ambiguous(candidates:)` ‚Äî never a silent first pick.
- [ ] Ladder precedence holds: content containing both a verbatim occurrence and a confusable variant resolves at `.exact`, not `.unicode`.
- [ ] All existing golden vectors in `Tests/FileToolTests/Fixtures/edit-match-golden.json` pass unchanged.
- [ ] A near-miss whose diff contains a confusable-equal expected/actual line pair carries a `note` naming the code point; a near-miss without confusable differences carries `note == nil`.
- [ ] `EditFile` output for such a near-miss includes the note on the `EditNearMiss` wire entry.
