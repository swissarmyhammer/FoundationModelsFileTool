## What

Port grok/codex apply_patch's pass-4 matcher behavior (see `xai-org/grok-build`, `seek_sequence.rs`) into our recovery ladder: an ASCII-authored `find` must match source text containing typographic punctuation (en/em dashes, smart quotes, exotic spaces) — the single most common real-world edit-match failure. Two facets, one concern: fold confusables when matching, and name confusables when a near-miss diff still fails.

**1. `Sources/FileTool/EditMatch.swift` — new ladder rung**

- Add a static `foldConfusables(_ text: String) -> String` that maps, per Unicode scalar:
  - dashes `U+2010`–`U+2015`, `U+2212` → `-`
  - single quotes `U+2018`–`U+201B` → `'`
  - double quotes `U+201C`–`U+201F` → `\"`
  - spaces `U+00A0`, `U+2000`–`U+200A`, `U+202F`, `U+205F`, `U+3000` → space
  - everything else unchanged. Make it `package`/`public` visible — `EditEngine` needs it for the diagnostic facet.
- Add `Rung.unicode`. Ladder order becomes exact → normalized → **unicode** → anchor → fuzzy (order is the contract; the new rung must only fire when `normalized` has descended).
- Implement the rung by generalizing the existing `locateLineBlock` line-comparison machinery to take a line normalizer, so the unicode rung is `locateLineBlock` with `normalize ∘ foldConfusables` — do not duplicate the window-scan loop. Uniqueness/ambiguity flows through the existing `finalizeBlockMatches` (one match → `.unique(range:rung:confidence: 1.0)`, several → `.ambiguous`, none → descend).
- Do **not** change the exact, normalized, anchor, or fuzzy rungs (fuzzy scoring stays confusable-blind): the Rust-parity golden vectors in `Tests/FileToolTests/Fixtures/edit-match-golden.json` must pass byte-identical. Update the `EditMatch` header doc: the ladder is now five rungs, and the unicode rung is a deliberate extension beyond the Rust `swissarmyhammer-edit-match` crate (flag for future upstream parity).

**2. `Sources/FileTool/EditEngine.swift` — confusable diagnostic in near-miss diffs**

When the ladder still fails (the span differs by more than punctuation), the near-miss diff currently shows a smart-quote line and an ASCII-quote line as an opaque expected/actual pair. In the diff construction (`diff(expected:actual:)` / `nearMiss(for:span:)`): when an `expected` line and an `actual` line differ but are equal after `normalize` + `EditMatch.foldConfusables`, surface a diagnostic. Add an optional `note: String?` to `EditEngine.NearMiss` (default `nil`) naming the first confusable difference with its code point, e.g. `differs only by Unicode punctuation: the file has '\u{2019}' (U+2019) where the find has \"'\" (U+0027)`.

**3. `Sources/FileTool/Operations/EditFile.swift` — wire projection**

Project the new note onto `EditNearMiss` as an optional `note` field (nil-omitted like the other optional outcome fields). No status/`matchedBy` changes: a unicode-rung match already surfaces as `recovered` via the existing `EditEngine.Resolution` mapping.
